Step 1: Introduction
What is information security and what are CIA security properties?

Information Security is the degree of resistance to / protection from harm to a valuable asset, which could be code, user data, or information.

CIA Properties:

  • Confidentiality: Information is not made available or disclosed to unauthorised entities.
    In the context of firmware, this means not giving access to firmware to any external entity by blocking flash read operations.
    Though it is not a part of secure boot, this avoids copying of firmware images by unknown sources.
  • Integrity: Maintaining and ensuring the accuracy and completeness of data. Data cannot be modified in an unauthorised or undetected manner.
  • Availability: Information is available when needed. Computing systems store and process information,
    security controls are used to protect it, and communication channels are used to access it.
Why the need for secure boot?

Secure boot is needed to ensure an embedded device only runs authentic, untampered firmware from power-on, blocking persistent malware and unauthorized changes that could compromise safety, data, or networks. It is more important in devices which connect to the internet and support over-the-air updates, to avoid malicious attacks and unauthorized access to devices.

Step 2: Prerequisites
Hardware must support secure boot, usually flash protection!

To make the boot loader secure, we must have a hashing algorithm, such as SHA-256, ready to generate a firmware digest for the application image, and public and private keys to sign the firmware digest and metadata, using an algorithm like RSA-2048/256-bit or ECDSA.

Three Important Parts of Secure Boot:

1. Firmware Confidentiality: This is achieved by making the flash read-protected, so it cannot be read by any external entity. This is particularly used for the bootloader to prevent unauthorized access to the firmware code.

2. Firmware Integrity: This is achieved by hashing the firmware image using SHA-256 algorithms to generate the hash/firmware digest, which is then stored at a secure location. This ensures that any tampering with the firmware can be detected.

3. Firmware Authenticity: This is achieved by signing the image and then verifying the signature before starting the application, using algorithms like RSA or ECDSA. This process uses private and public keys, with the public key stored at a fixed flash address to be used by the bootloader to verify the signature.

Step 3: Configuration
Flash memory layout configuration:

To set up secure boot, the bootloader and the application must be built and stored separately. Reserve two dedicated flash regions: one for the bootloader at the beginning of flash, and another for the application. Additionally, reserve a fixed flash address for the public key used for signature verification‚Äîcommonly placed immediately after the bootloader and before the application. This public key region can be at any fixed flash location, but it must be protected, immutable, and treated as secure.

Setting Up Secure Boot

First, we have to secure the flash region where the boot loader is located, so that this memory cannot be accessed by anything other than the bootloader code. On STM microcontrollers, this can be achieved using the secure flash page option. Additionally, we must make the bootloader flash memory region write-protected, so that it cannot be overwritten or erased.
